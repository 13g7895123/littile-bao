## 一、背景

`2026-06-05` 上週五的盤中資料顯示，目前買進邏輯有兩類問題：

1. 已經鎖板，但畫面或成交紀錄晚了 `3` 秒以上才出現買進。
2. 有些標的曾符合漲停條件，但最終沒有買進，尤其是短暫鎖板情境。

這代表目前系統不是單純「測試單成交時間比較晚」，而是整體進場判斷鏈路存在：

- 行情時間與本機時間混用
- 買進判斷粒度過粗
- 缺少完整監控池留痕

### 2.1 行情事件本身已有明顯延遲

錄製檔中的 `tick` / `book` 都同時保留：

- `api_time`
- `recv_time`
- `latency_ms`

實際觀察可見，`recv_time - api_time` 大多約為 `2.8` 秒。

這表示：

- 就算 API 事件在 `09:32:24` 發生
- 程式實際收到並處理，很可能已接近 `09:32:27`

### 2.2 封板狀態是事件驅動更新，但買進不是

目前 `TradingEngine`：

- 在 `_on_tick()` / `_on_book()` 內即時更新 `is_at_limit_up`
- 但真正的進場邏輯在每秒一次的 `_tick()` 裡才執行

也就是說：

- 「是否鎖板」是逐筆事件更新
- 「要不要下單」仍是每秒輪詢

### 2.3 已發生的秒差案例可被重現解釋

- `5202`：模式成立到 `PLACE` 約差 `3.3` 秒
- `6432`：模式成立到 `PLACE` 約差 `3.8` 秒

這與：

- `~2.8` 秒行情接收延遲
- 再加上下一輪 `_tick()` 等待

是相符的。

### 2.4 短暫鎖板會被漏掉

以 `6763` 為例：

- 曾短暫出現封板
- 很快又打開
- 若下一次 `_tick()` 執行時已不在 `is_at_limit_up=True`，就不會買進

這是輪詢架構的典型漏單情境。

### 2.5 部分未買標的目前缺少可追查證據

像 `5426`、`4939`、`3441`：

- 在 `program.log` 與 `dry_run_audit` 中沒有對應進場紀錄
- 但目前沒有一份完整的盤中監控池歷史快照，可證明它們當下是否在池內

因此目前仍缺「池內 / 池外」的證據鏈。

## 三、問題定義

本次要修的不是單一 bug，而是三個互相關聯的設計問題。

### 問題 A：買進判斷時間落後市場事件

現況：

- 封板訊號已在 `_on_tick()` / `_on_book()` 到達
- 但要等 `_tick()` 每秒輪詢才真正送單

結果：

- 容易出現 `3` 秒以上秒差

### 問題 B：短暫鎖板漏單

現況：

- 若某檔只鎖幾百毫秒到 `1` 秒內
- `_tick()` 執行時可能已經解鎖

結果：

- 盤中曾符合條件，但實際沒買

### 問題 C：監控池缺少完整留痕

現況：

- `PoolSwap` 會動態換股
- 但沒有保存每次換股後的完整池內快照

結果：

- 事後很難證明某檔當下到底有沒有在 500 檔池內

## 四、根因整理

### 4.1 秒差的直接根因

1. `FubonRealtimeFeed` 收到行情時，`recv_time` 已晚於 `api_time`
2. `TradingEngine` 買進邏輯仍依賴 `_tick()` 每秒輪詢
3. `dry_run` 又有自己的 `PLACE/FILL` 時間

### 4.2 漏單的直接根因

1. `is_at_limit_up` 雖然已逐筆更新
2. 但沒有在「首次封板成立」的當下立即重跑進場判斷
3. 短暫封板因此可能被整段錯過

### 4.3 無法追查的直接根因

1. `PoolSwap` 只有 log 摘要
2. 沒有盤中每輪的完整清單快照
3. 復盤時無法回答「某檔當時到底在不在池內」

---

## 五、修正目標

### 5.1 進場時效目標

- 封板訊號首次成立時，不必等下一秒輪詢才判斷是否下單
- 正常情況下，送單時間應更貼近事件到達時間

### 5.2 漏單降低目標

- 對於短暫鎖板，至少要在首次成立時有一次即時進場機會

### 5.3 追查能力目標

- 能事後回答：
  - 某檔是否在池內
  - 某檔何時首次符合模式
  - 某檔為何沒買

### 5.4 報表一致性目標

- 調查文件、GUI、log、audit 都能清楚區分：
  - 市場事件時間
  - 本機接收時間
  - 下單時間
  - 成交時間

## 七、修正方案

## 7.1 將買進判斷補成「事件驅動 + 每秒輪詢」雙軌

保留現有 `_tick()`，但新增一條即時路徑：

### 作法

- 當 `_refresh_limit_up_state()` 發現某檔從 `False -> True` 首次進入 `is_at_limit_up=True`
- 立即觸發一次進場判斷

建議新增方法：

- `_evaluate_entry_on_limit_up_event(state, event_time)`

這個方法應：

- 共用 `_tick()` 內既有的進場規則
- 但只針對單一標的立刻評估

### 預期效果

- 不用等下一秒輪詢
- 可大幅縮短 `5202` / `6432` 類型的秒差
- 可提高 `6763` 類型短暫鎖板的命中率

## 7.2 把進場條件抽成單一共用函式

目前買進判斷主要寫在 `_tick()` 裡。

建議拆成：

- `_evaluate_entry(state, now, trigger_source)`

其中：

- `_tick()` 呼叫它
- 封板事件即時路徑也呼叫它

### 好處

1. 避免複製邏輯
2. 確保事件驅動與輪詢邏輯一致
3. 測試更容易覆蓋

## 7.3 明確記錄四種時間

目前調查時需要人工比對 `api_time`、`recv_time`、`program.log`、`audit ts`。

建議在 decision / strategy event 中固定加上：

- `market_event_time`
- `recv_time`
- `decision_time`
- `order_place_time`

若是 dry-run fill，再保留：

- `fill_time`

### 好處

- 以後可以直接從同一條紀錄看出時間差來源
- 不必再手動交叉比對多份檔案

## 7.4 增加「未買原因」的事件級留痕

目前 `ENTRY_SKIP` 有原因，但缺少完整上下文。

建議當略過進場時，多保留：

- `trigger_source`：`tick_loop` 或 `limit_up_event`
- `market_event_time`
- `is_at_limit_up`
- `candle`
- `ask_qty`
- `last_1s_vol`
- `pool_membership_version`

### 好處

- 能直接判斷：
  - 是時間超過
  - 是短暫鎖板消失
  - 還是當下根本不在池內

## 7.5 為 `PoolSwap` 建立完整快照留痕

目前 log 只有：

- 池內幾支
- 換出哪些
- 補入哪些

建議每次換股後，額外落一份 machine-readable 資料，例如：

- `pool_snapshots_YYYYMMDD.jsonl`

每筆至少包含：

- `ts`
- `pool_version`
- `active_codes`
- `added_codes`
- `removed_codes`
- `reason`

### 好處

- 事後可明確回答 `5426 / 4939 / 3441` 是否真的在池內

## 7.6 補一個「當日未買候選報表」

盤後可依 recording + strategy event 產出：

- 曾觸發封板
- 但未成交買進

並列出未買原因：

- `已過進場時段`
- `短暫鎖板未命中`
- `不在池內`
- `資金不足`
- `特殊股排除`

### 好處

- 減少人工追 log 成本
- 更適合用來核對「照理應該符合很多」的盤感

## 十、測試計畫

### 單元測試

1. 封板首次成立時，事件驅動路徑可立即下單

## 十二、風險與對策

### 風險 1：事件驅動後可能重複下單

原因：

- tick 與 book 可能連續多次都顯示鎖板

對策：

- 只在 `False -> True` 的狀態轉換點觸發
- 並保留 `pending` 防重複

### 風險 2：事件驅動與輪詢結果不一致

原因：

- 若兩條路徑各自有不同條件實作

對策：

- 強制共用同一個 `_evaluate_entry(...)`

### 風險 3：log / event 欄位膨脹

原因：

- 補太多時間與上下文字段

對策：

- 先加在 decision event 與診斷檔
- GUI 僅顯示必要欄位

### 風險 4：快照檔案太大

原因：

- 每次換股都寫 500 檔完整清單

對策：

- 採 `jsonl`
- 保留最近 `N` 天
- 必要時只對差異與版本做壓縮存放

1. 封板首次成立時，可立即觸發一次進場評估，不再完全依賴每秒輪詢

5. 不發生重複下單、重複成交紀錄或既有策略回歸
