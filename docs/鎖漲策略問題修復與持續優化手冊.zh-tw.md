# 鎖漲策略問題、修復與持續優化手冊

> 最後更新：2026-06-11
> 適用範圍：`build_win/src/limitup_detection.py`、`build_win/src/engine.py`、行情錄製、離線分析、進場決策與相關測試

## 1. 文件目的

本文件集中記錄鎖漲策略已發生的問題、修復演進、判斷定義、證據來源、驗證方法與後續優化規則。

核心目標不是讓某一檔股票的時間看起來正確，而是建立可跨股票、跨交易日重複驗證的判斷方式，降低：

- 提前誤判：市場尚未真正鎖住，程式已認定鎖板並買進。
- 延後判斷：市場已鎖住，但程式數秒或數分鐘後才確認。
- 短暫鎖板漏單：符合條件的時間太短，被每秒輪詢錯過。
- 無法追查：事後只有人工觀察時間，缺少原始行情、決策與下單證據。
- 單一案例過度擬合：修好一檔，卻使另一檔判斷錯誤。

## 2. 必須區分的時間

調查時不可把以下時間混稱為「買進時間」或「鎖漲時間」：

| 名稱 | 定義 | 主要來源 |
|---|---|---|
| 人工真實鎖漲時間 | 使用者依看盤軟體確認真正鎖住的時間，作為標註基準 | 人工觀察、截圖、外部看盤紀錄 |
| 候選鎖板時間 | 任一寬鬆訊號首次顯示可能鎖板 | `LIMIT_UP_CANDIDATE`、錄盤回放 |
| 正式確認時間 | 目前正式模式由 `False -> True` 的市場事件時間 | `confirmed_lock_event_time`、`LIMIT_UP` |
| 本機接收時間 | 程式收到行情的時間 | `recv_time` |
| 策略決策時間 | 引擎完成進場判斷的時間 | `decision_events` |
| 委託送出時間 | 程式呼叫券商下單的時間 | `decision_events`、audit、券商委託紀錄 |
| 成交時間 | 券商回報成交的時間 | 成交回報、成交紀錄 |

比較準確度時，以「正式確認市場事件時間」對「人工真實鎖漲時間」；比較系統延遲時，才分析市場事件、本機接收、策略決策、委託送出的時間差。

## 3. 行情資料特性

### 3.1 Tick 與 Book 是兩條非同步資料流

- `tick` 表示逐筆成交，包含最新成交價、成交量、`isLimitUpPrice`、`isLimitUpBid` 等旗標。
- `book` 表示五檔委買委賣，包含 `bid[]`、`ask[]`。
- 兩者到達順序不保證一致。
- 引擎會保存兩條資料流的最近狀態，再合併判斷。

因此，單純把「最近一次 tick」和「最新 book」做布林 AND，可能把不同市場階段的訊號錯誤拼接。

### 3.2 富邦五檔可能出現 `price=0` 佔位檔

實盤曾出現：

- `bid[0].price = 0`
- `bid[1]` 或後續檔位才是真正有效買價
- `ask = []`

若只讀原始 `bid[0]`，會把已經存在的漲停買盤判成不存在。現在引擎會略過價格小於等於零的買盤，取得 `effective_bid0_price / effective_bid0_volume`。

### 3.3 空委賣不等於資料永遠可靠

`ask=[]`、`ask1.size=0`、`ask1.price>limit_up` 都可能代表漲停價委賣已被吃完，但仍必須搭配有效買盤與新 tick 確認，不能只靠單一欄位判斷。

## 4. 判斷模式演進

### 4.1 寬鬆模式：`bid_or_trade_flag`

條件：

- 原始 `bid1` 在漲停價，或
- API `isLimitUpBid=true`

問題：

- 2026-05-26 實盤回放顯示大量命中時，五檔買一其實不在漲停價。
- 容易把觸板、短暫旗標或不完整鎖板認定為正式鎖板。

### 4.2 第一版嚴格模式：`strict_lock_from_user_rule`

條件：

- `isLimitUpPrice=true`
- 原始 `bid[0]` 在漲停且有量
- 無委賣、賣一量為零，或賣一高於漲停

改善：

- 大幅降低只有 API 旗標、但五檔未鎖漲的誤判。

問題：

- 原始 `bid[0]` 可能是 `price=0` 佔位檔，造成真正鎖板後仍延遲確認。

### 4.3 第二版：`strict_lock_with_effective_bid`

條件：

- `isLimitUpPrice=true`
- 第一個價格大於零的有效買盤在漲停且有量
- 無委賣、賣一量為零，或賣一高於漲停

改善：

- 修正 `bid[0]=0` 佔位問題。
- 可更早識別 6174 類型的真實鎖板。

問題：

- Book 先進入鎖板形態、tick 尚未正式確認時，可能過早成立。

### 4.4 第三版：`strict_lock_with_effective_bid_tick_confirmed`

正式規則：

1. 有效買盤在漲停且有量。
2. 無委賣、賣一量為零，或賣一高於漲停。
3. 上述條件形成一段新的「有效五檔鎖板候選段」。
4. 候選段成立後，必須再收到新的 tick，且：
   - `isLimitUpPrice=true`
   - `isLimitUpBid=true`
5. 候選段成立前留下的舊 tick 旗標不得用來確認新候選段。
6. 候選條件解除時，清除該段確認狀態；下一段必須重新確認。

這是目前正式預設模式。

## 5. 已發生問題與修復

### 5.1 2026-05-26：寬鬆模式大量誤判

證據：

- 實盤錄製檔共讀取 6,391,373 行。
- `bid_or_trade_flag` 命中 49,726 次。
- `strict_lock_from_user_rule` 命中 2,540 次。
- 寬鬆模式被擋下的主因中，`bid1 未在漲停價` 共 47,133 次。

根因：

- API 旗標或 trade bid 訊號存在，不代表五檔已形成完整鎖板。

修復：

- 導入嚴格模式，要求成交旗標、買盤與委賣狀態同時成立。

### 5.2 2026-06-05：買進秒差與短暫鎖板漏單

問題：

- 行情到達後，進場曾仍依賴每秒輪詢。
- 市場事件到本機已有延遲，再等待下一輪輪詢，買進可再晚約一秒。
- 短暫鎖板可能在輪詢前已打開，因此完全漏掉。

修復：

- `_refresh_limit_up_state()` 偵測正式鎖板由 `False -> True` 時，立即執行一次 `_evaluate_entry()`。
- 保留每秒輪詢作為補充，但不再只依賴輪詢。
- 增加候選時間、正式確認時間與決策事件留痕。

### 5.3 2026-06-11：6174 因 `bid[0]=0` 延後確認

已知時間：

| 項目 | 時間 |
|---|---:|
| 第二版有效買盤候選首次成立 | 約 `10:01:42.861128` |
| 人工真實鎖漲時間 | `10:01:47.340631` |
| 舊版原始 `bid[0]` 嚴格規則首次成立 | 約 `10:05:18.157921` |
| 第三版優化後正式確認 | `10:01:47.340631` |

根因：

- 舊版只讀 `bid[0]`，被價格為零的佔位檔遮蔽真正買盤。

修復：

- 從 `bid[]` 找第一個價格大於零的有效買盤。
- 再等待 tick 正式確認，避免只有 book 先動就買進。

### 5.4 2026-06-11：6156 因舊 tick 旗標被沿用而提前確認

已知時間：

| 項目 | 時間 |
|---|---:|
| 人工真實鎖漲時間 | `11:05:52` |
| 修正前程式實際買入 / 判斷時間 | 目前工作區缺少當日成交與決策檔，不能可靠填寫 |
| 狀態式確認優化後 | `11:05:52` 對應的新確認 tick |

根因：

- 初版第三模式只檢查合併後狀態是否同時滿足：
  - 有效買盤鎖漲
  - 最近一次 tick 的 `isLimitUpPrice=true`
  - 最近一次 tick 的 `isLimitUpBid=true`
- 若 true 旗標來自候選段成立前，新的 book 到達時仍可能立刻被確認。

修復：

- `StockState` 新增：
  - `effective_lock_segment_active`
  - `effective_lock_segment_tick_confirmed`
- 新候選段開始時清除舊 tick 確認。
- 只有 `source == "tick"` 且該段內新 tick 同時帶兩個 true 旗標，才把正式模式設為 true。
- 每日重置時同步清除此狀態。

## 6. 目前程式責任分界

| 模組 | 責任 |
|---|---|
| `limitup_detection.py` | 將行情快照轉成原子 signals 與無狀態候選模式 |
| `engine.py::_on_tick/_on_book` | 更新最新 tick/book 狀態 |
| `engine.py::_refresh_limit_up_state` | 管理跨事件、跨候選段的狀態式確認 |
| `broker/recording.py` | 非阻塞錄製原始與結構化行情 |
| `analyze_limitup_logs.py` | 比較無狀態 signals / candidates 首次成立時間 |
| `replay_limitup_trace.py` | 回放指定模式狀態切換 |
| `decision_events` | 保存候選、確認、略過、封鎖與策略決策證據 |

重要限制：

- `strict_lock_with_effective_bid_tick_confirmed` 的正式含義包含「候選段成立後的新 tick」，是引擎狀態式規則。
- 離線分析器若只直接呼叫 `evaluate_limit_up_state()`，不一定完整重現引擎的候選段確認語意。
- 後續修改正式模式時，必須同步檢查分析器與回放器是否和引擎等價；不等價時，不得用分析器輸出宣稱正式策略時間。

## 7. 證據優先級

調查一律依下列順序取證：

1. 當天 `session_*.ticks.ndjson.gz` 與對應 `meta.json`
2. `decision_events.YYYYMMDD.jsonl`
3. `dry_run_audit_YYYYMMDD.jsonl` 或券商委託 / 成交紀錄
4. `program.log.YYYYMMDD`
5. 人工觀察、截圖與口述時間

人工時間是必要標註，但不能單獨證明程式實際買入時間。缺少證據時必須明確寫「無法可靠確認」，不可猜測。

## 8. 每次問題調查流程

1. 明確列出股票代碼、日期、人工真實鎖漲時間與問題類型。
2. 找出當天錄盤、meta、decision events、audit 與 program log。
3. 確認 `meta.json` 中當天實跑的設定與模式，不能用目前預設值代替。
4. 對單檔回放並列出：
   - 原子訊號首次時間
   - 各模式首次成立與每段切換
   - 候選段開始、解除與新 tick 確認時間
   - 正式確認、決策、委託、成交時間
5. 將誤差拆成：
   - 判斷規則誤差
   - 市場到本機接收延遲
   - 接收到決策延遲
   - 決策到委託延遲
   - 委託到成交延遲
6. 先寫可失敗的回歸測試，再修改正式邏輯。
7. 用問題股、已修正股及正常對照股共同驗證。
8. 更新本文件、對應 `docs/skills/` 與測試。

## 9. 修改規則

### 必須做

- 用事件順序與候選段建模，避免只增加更多快照布林條件。
- 對 `price=0`、空陣列、缺欄位、tick/book 亂序、舊旗標殘留建立測試。
- 確保正式引擎、分析器、回放器對同一模式的語意一致，或清楚標註差異。
- 至少回歸：
  - 原問題股
  - 前一次修好的股票
  - 一檔正常鎖板對照
  - 一檔觸板但未鎖的負樣本
- 執行完整測試。

### 禁止做

- 不得為單一股票代碼寫例外。
- 不得只依人工時間硬調固定延遲秒數。
- 不得把 API 旗標當成官方「正式鎖板」單一真相。
- 不得把分析器首次命中時間直接當作券商委託或成交時間。
- 不得在缺少當日證據時填入推測的買進時間。

## 10. 驗收標準

一個鎖漲修復可以接受，至少需滿足：

- 問題可由原始事件序列或測試重現。
- 修復前測試失敗，修復後測試成功。
- 原問題股的正式確認時間對齊人工標註，或差異有可量化原因。
- 前一次修好的股票沒有回歸。
- 負樣本不會被提前確認。
- 候選、確認、決策與下單時間可以分別追查。
- 完整測試通過。
