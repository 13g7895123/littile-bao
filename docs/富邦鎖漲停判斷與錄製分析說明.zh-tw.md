# 富邦鎖漲停判斷與錄製分析說明

本文說明本專案已補上的「鎖漲停多模式判斷」、「行情錄製欄位」與「離線分析程式」。

## 目的

你要區分的不只是「觸及漲停」，而是更細的幾種盤中狀態：

- `觸及漲停`：最新成交價到達漲停價。
- `部分封板`：賣一仍在漲停價，表示漲停價還有委賣排隊。
- `完全鎖板`：買一已經在漲停價，而且委賣可能已被吃光。
- `鎖板打開`：原本封板，後續賣一掉回漲停價以下，或買一不再是漲停價。

因為富邦公開行情沒有「這一刻正式鎖板」的單一官方旗標，所以本次改成：

1. 主程式先啟用第 1 種候選邏輯。
2. 同時把足夠的行情欄位錄下來。
3. 透過離線分析程式回推每一種候選邏輯的首次觸發時間與次數。

## 目前已啟用的判斷模式

設定檔鍵值：

- `limit_up_detection_mode`

目前預設值：

- `strict_lock_from_user_rule`

代表：

- `isLimitUpPrice == true`
- 且 `bid1 == 漲停價`
- 且 `bid1.size > 0`
- 且 `asks` 為空，或 `ask1.size == 0`，或 `ask1.price > limitUpPrice`

這是目前較嚴格的「真正鎖板」定義，用來避免把單純 `isLimitUpBid=true` 或僅有最新成交打到漲停價的情況誤判成完全鎖板。

## 所有候選模式

程式位置：

- [limitup_detection.py](/home/jarvis/project/bonus/17_littile-bao/build_win/src/limitup_detection.py:1)

已實作的候選模式如下：

1. `ask_or_bid_or_last`
   `ask1=漲停` 或 `bid1=漲停` 或 `最新成交=漲停`
2. `ask_only`
   只有 `ask1=漲停`
3. `bid_only`
   只有 `bid1=漲停`
4. `bid_or_trade_flag`
   `bid1=漲停` 或富邦旗標 `isLimitUpBid=true`
5. `bid_and_last`
   `bid1=漲停` 且 `最新成交=漲停`
6. `bid_and_no_ask`
   `bid1=漲停` 且五檔完全沒有委賣
7. `bid_and_zero_ask`
   `bid1=漲停` 且沒有委賣，或賣一量為 0
8. `strict_lock_from_user_rule`
   `isLimitUpPrice=true` 且 `bid1=漲停且有量`，且 `asks` 為空 / `ask1.size=0` / `ask1.price>limitUpPrice`
9. `trade_price_only`
   只有 `最新成交=漲停`
10. `trade_flag_only`
   只有富邦漲停旗標為真

## 已錄製的關鍵欄位

錄製檔仍沿用原本 `log/recordings/<YYYYMMDD>/session_<HHMMSS>.ticks.ndjson.gz`。

### `tick` 紀錄

除了原本欄位外，新增：

- `bid`
- `ask`
- `is_limit_up_price`
- `is_limit_up_bid`
- `is_limit_up_ask`
- `is_trial`

用途：

- 判斷最後成交當時是否貼著買價或賣價
- 比對富邦是否直接提供漲停旗標
- 區分試撮資料

### `book` 紀錄

保留：

- `ask[]`
- `bid[]`

用途：

- 判斷 `ask1` 是否在漲停
- 判斷 `bid1` 是否在漲停
- 判斷是否完全無委賣

### `meta.json`

保留每次錄製的：

- `config_snapshot`
- `symbol_universe`
- `symbol_count`

其中 `symbol_universe` 內已有：

- `code`
- `name`
- `market`
- `prev_close`
- `limit_up`

離線分析程式會用這個 `limit_up` 當作每檔股票的漲停基準。

## 離線分析程式

程式位置：

- [analyze_limitup_logs.py](/home/jarvis/project/bonus/17_littile-bao/build_win/src/analyze_limitup_logs.py:1)

### 用法

分析整個錄製檔：

```bash
python3 build_win/src/analyze_limitup_logs.py \
  build_win/src/log/recordings/20260520/session_090001.ticks.ndjson.gz
```

只分析單一股票：

```bash
python3 build_win/src/analyze_limitup_logs.py \
  build_win/src/log/recordings/20260520/session_090001.ticks.ndjson.gz \
  --code 2330
```

### 輸出內容

每檔股票會列出兩大區塊：

- `signals`
  各個原子訊號第一次出現的時間
- `candidates`
  每個候選模式第一次成立的時間與成立次數

其中最重要的 `signals` 包含：

- `ask_at_limit`
- `bid_at_limit`
- `last_at_limit`
- `trade_flag_price`
- `trade_flag_bid`
- `trade_flag_ask`
- `ask_empty`
- `ask_qty_zero`
- `trade_at_ask`
- `trade_at_bid`

## 注意事項

- `bid1=漲停` 比 `最新成交=漲停` 更接近「真的鎖板」。
- `ask1=漲停` 常代表「有封板，但漲停價仍有委賣排隊」。
- `ask` 為空或 `ask1` 不存在，不會直接由 API 明講成「0 張」，需由程式自行解讀。
- `tradeVolumeAtAsk / tradeVolumeAtBid` 這類內外盤累計成交量，本次沒有列為主判斷，只建議作輔助觀察。
