## C 段目前為什麼慢

### Dry-run

`DryRunOrderManager` 目前流程：

1. `place_order()`
2. 立刻 dispatch `OrderStatus.PENDING`
3. `random.uniform(min_delay, max_delay)`
4. `threading.Timer(delay, _simulate_fill)`
5. 到時間後才 dispatch：
   - `OrderStatus.FILLED`
   - `FillEvent`

這段天生就會產生延遲。

### Mock

`MockOrderManager` 也是同樣概念：

- 立即送 `PENDING`
- 之後用 `Timer` 延遲送 `FILLED` / `FillEvent`

只是它的預設區間是：

- `fill_delay_range=(0.6, 1.8)`

所以 `Mock` 模式甚至比 `Dry-run` 還可能更慢。

## C 段如果要在 MOCK / DRY 模式下「最快」，有兩種做法

## 做法一：不改碼，直接把延遲設成 0

這是最小成本、最安全、最容易落地的做法。

### Dry-run

把：

- `dry_run_fill_min_sec = 0.0`
- `dry_run_fill_max_sec = 0.0`

就會變成：

- 仍先送 `PENDING`
- 但 `Timer(0)` 幾乎會立刻觸發 `FILLED`

### 優點

- 不用改核心邏輯
- 不影響 audit log 格式
- 不影響 GUI 對 `PENDING -> FILLED` 狀態流的相容性
- 最適合先拿來做低延遲驗證

### 缺點

- 雖然非常快，但仍然經過一次 `Timer` / 執行緒切換
- 不是理論上的最短路徑

### 適用情境

- 想保留完整委託生命週期
- 想保留「先掛單、再成交」的 UI 表現
- 但不想再被 0.5~1.5 秒的模擬延遲污染測試結果

---

## 做法二：改碼，提供「立即成交模式」

這是 `MOCK / DRY` 模式下真正的最速方案。

### 核心思路

當設定為「立即成交」時：

1. `place_order()`
2. 立刻 dispatch `PENDING`
3. 同一個 call stack 內直接執行 `_simulate_fill()`
4. 立即 dispatch：
   - `FILLED`
   - `FillEvent`

也就是完全不經過：

- `random.uniform()`
- `threading.Timer`
- 額外的背景執行緒排程等待

### 建議實作

新增一個布林設定，例如：

```python
dry_run_instant_fill: bool = False
mock_instant_fill: bool = False
```

然後在 `DryRunOrderManager.place_order()`：

```python
if self.instant_fill:
    self._simulate_fill(order_id)
else:
    delay = random.uniform(*self.fill_delay_range)
    timer = threading.Timer(delay, self._simulate_fill, args=(order_id,))
    timer.daemon = True
    timer.start()
```

`MockOrderManager` 同理。

### 優點

- 這是 `MOCK / DRY` 下最短的 `C 段`
- 可以最大程度隔離「模擬成交器」本身的人為延遲
- 更適合做：
  - 策略判斷延遲驗證
  - UI 端到端延遲驗證
  - 事件鏈路壓測

### 缺點

- 會讓 `PENDING -> FILLED` 幾乎瞬間完成
- 有些依賴「掛單中」狀態停留時間的測試，可能要重新調整
- 若 callback 內有重入風險，需確認不會造成邏輯互踩

### 適用情境

- 你要的是「最快」
- 不是要模擬真實券商排隊或成交等待

## 建議的優先順序

### 第一階段

先不改碼，直接把 `Dry-run` 延遲調成 `0`

建議值：

```json
{
  "dry_run_fill_min_sec": 0.0,
  "dry_run_fill_max_sec": 0.0
}
```

目的：

- 快速驗證 `C 段` 移除後，整體延遲是否主要只剩 `A + B`
- 幾乎零風險

### 第二階段

若你要更極端的低延遲測試，再加「立即成交模式」

建議：

- `DryRunOrderManager` 加 `instant_fill`
- `MockOrderManager` 加 `instant_fill`
- 預設仍保持 `False`
- 只有在性能驗證或特定自動化測試時開啟

## 實務結論

如果目標是「讓 MOCK / DRY 模式下也能最快速」：

### 最務實答案

先做這件事就夠了：

- 把 `dry_run_fill_min_sec` 和 `dry_run_fill_max_sec` 都設成 `0`

這能立刻把今天 `C 段` 那種 `0.8 ~ 1.3 秒` 的人為延遲幾乎全部拿掉。

### 最徹底答案

若要理論最短：

- 新增 `instant_fill` 模式
- 直接同步呼叫 `_simulate_fill()`
- 完全繞過 `Timer`

---

## 最後建議

優先順序應該是：

1. 先保留本次已完成的 `B 段` 即時判斷修正
2. 把 `DRY` 的 `fill_delay_range` 調成 `0`
3. 重新量一次 `13:00` 後的三段延遲
4. 若仍要更快，再加 `instant_fill`

原因是：

- `B 段` 是真實策略反應能力
- `C 段` 在 `MOCK / DRY` 下目前大多只是模擬器故意加的等待

所以不要先被 `C 段` 的數字誤導，以為市場或策略真的慢。
