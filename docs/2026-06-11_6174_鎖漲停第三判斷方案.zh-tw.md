# 2026-06-11 6174 鎖漲停第三判斷方案

## 背景

`6174 安碁` 在 `2026-06-11` 盤中有一段很典型的富邦五檔資料：

- `ask = []`
- `last_price = 51.7`
- `is_limit_up_price = true`
- `bid[0].price = 0`
- 後續有效買盤檔位已經出現 `51.7`

舊規則 `strict_lock_from_user_rule` 只看 `bid[0]`，因此會被 `price=0` 的佔位檔誤導，直到更晚 `bid[0]` 自己也變成 `51.7` 才認定正式鎖板。

第二版規則 `strict_lock_with_effective_bid` 已修正這點，會跳過 `bid[0]=0` 去找第一個有效買盤，因此可以更早辨識真正鎖板。

但如果目標是把「正式確認時間」盡量對齊你觀察到的 `10:01:47`，還需要再加上一層 tick 確認。

## 第三種模式

新增模式：

- `strict_lock_with_effective_bid_tick_confirmed`

判斷條件：

1. 第一個有效買盤 `effective bid1` 必須等於漲停價
2. `effective bid1` 的量必須大於 0
3. 同時滿足以下其中之一：
   - 沒有任何委賣檔
   - 賣一量為 0
   - 賣一價格高於漲停價
4. 上述五檔鎖板候選成立後，必須再收到新的 tick：
   - `isLimitUpPrice = true`
   - `isLimitUpBid = true`

這個模式與第二版規則的差別是：

- 第二版只確認當下合併後的行情欄位。
- 第三版會記住每一段新的五檔鎖板候選，且不允許沿用候選成立前留下的舊 tick 旗標。

這可避免非同步到達的 book 與舊 tick 被錯誤拼成一次正式鎖板。

## 為什麼把第三版設成預設

這次改成預設的理由是：

- 可以避開 `bid[0]=0` 的富邦佔位問題
- 不會像第二版那樣在只有 book 先動、tick 還沒確認時就提早判成正式鎖板
- 比較接近實際盤中人工觀察「確定鎖住」的時間點

## 6174 當天時間對照

以 `2026-06-11` 的錄盤為例：

- 第二版 `strict_lock_with_effective_bid`
  - 啟用後重新撬開再重鎖的首次成立時間：約 `10:01:42.861128`
- 第三版 `strict_lock_with_effective_bid_tick_confirmed`
  - 會落在該段五檔候選成立後，第一筆同時滿足 `isLimitUpPrice=true` 且 `isLimitUpBid=true` 的新 tick
  - 依原始錄盤，對應你觀察到的 `10:01:47.340631`
- 舊版 `strict_lock_from_user_rule`
  - 首次成立時間：約 `10:05:18.157921`

## 取捨

第三版不是「最早發現鎖板」的模式，而是「較晚但更穩、且更接近 tick 正式確認」的模式。

如果未來需求改成：

- 要最早發現重鎖：用 `strict_lock_with_effective_bid`
- 要正式進場以 tick 確認為準：用 `strict_lock_with_effective_bid_tick_confirmed`

目前系統預設已切到第三版，代表盤中正式鎖板判定偏向保守確認。
