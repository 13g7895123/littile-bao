# Skill：鎖漲策略持續優化

## 目的

用錄盤、決策事件、委託 / 成交紀錄與人工真實鎖漲標註，持續修正鎖漲判斷準確度，同時避免修好單一股票卻使其他股票回歸。

完整問題歷史、定義與驗收規則：

- `docs/鎖漲策略問題修復與持續優化手冊.zh-tw.md`

## 觸發情境

- 程式比真實鎖漲時間提早或延後買進。
- 股票真正鎖漲但程式漏判。
- tick 與 book 訊號不一致。
- 修改、新增或切換 `limit_up_detection_mode`。
- 分析器 / 回放器時間與正式引擎不同。

## 必走流程

1. 列出日期、股票代碼、人工真實鎖漲時間與問題類型。
2. 取得當天錄盤、meta、decision events、audit / 券商紀錄與 program log。
3. 從 `meta.json` 或執行紀錄確認當天實際模式，不使用目前預設值代替。
4. 分開比較候選、正式確認、本機接收、決策、委託與成交時間。
5. 依原始事件順序重現 tick/book 非同步狀態。
6. 先建立會失敗的回歸測試，再修改正式策略。
7. 用問題股、歷史修復股、正常正樣本與未鎖負樣本共同回歸。
8. 執行完整測試並更新相關 skill 與手冊。

## 設計原則

- 無狀態 signals / candidates 放在 `limitup_detection.py`。
- 跨事件、候選段與重新確認狀態放在 `engine.py`。
- 新候選段不得沿用候選成立前的舊 tick 旗標。
- 遇到 `bid[0].price=0` 時，以第一個價格大於零的有效買盤判斷。
- 不為股票代碼寫例外，不以固定延遲硬對齊人工時間。
- 缺少程式買入或成交證據時，明確標示無法確認，不可猜測。
- 分析器與正式引擎語意不一致時，不得把分析器首次命中稱為正式確認時間。

## 最低回歸矩陣

| 類型 | 最少樣本 |
|---|---|
| 本次問題 | 所有回報股票 |
| 歷史修復 | 6174 與同類歷史股票 |
| 正樣本 | 一檔正常完整鎖板 |
| 負樣本 | 一檔觸板但未完整鎖板 |
| 事件順序 | 一組舊 tick 在前、新 book 候選在後的序列 |

## 驗收輸出

每次調查至少產出：

| 股票 | 程式判斷 / 買入時間 | 真實鎖漲時間 | 優化後時間 | 根因 | 證據 |
|---|---|---|---|---|---|

並附上：

- 修改前後行為
- 新增回歸測試
- 原問題與歷史案例驗證結果
- 完整測試結果
- 尚缺證據與下一次錄盤需求
