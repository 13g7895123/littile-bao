# Skill：開發與維護守則

> **適用對象**：所有改動 `build_win/src/` 內容的開發者或 AI 助理。
> **核心原則**：「先讀 skills → 再動程式碼 → 再回頭同步 skills」。

---

## 1. 開發 / 維護必走流程

### 1.1 接手任務時
1. **必先閱讀 `docs/skills/README.md`** 與該任務相關的物件 skill。
2. 若任務牽涉到多個模組（例如同時改 engine 與 gui），把所有相關 skill 一次讀完，再規劃改動。
3. 對照 skill 中描述的「輸入 / 輸出 / 與其他模組互動」，先確認你的修改是否會破壞既定介面。

### 1.2 撰寫程式前
1. 找出待修改的物件 / 函式對應的 skill 檔。
2. **如果 skill 與現況不符，先修 skill，再開始改程式碼**：避免 skill 文件落後於真實程式碼。
3. 如果新增功能在 skill 中不存在：
   - 先在 skill 內新增「擬新增的職責 / API / 與其他模組關係」段落。
   - 若整個物件因擴充而過於肥大，**建立新的模組子資料夾**並拆分 skill。
4. 若調整會影響其他模組的 skill（例如改變 broker 對 engine 的介面），同步更新相關 skill。

### 1.3 完成程式碼後
1. **回頭逐條校對對應 skill**，確保下列項目仍然準確：
   - 主要類別 / 函式列表
   - 對外公開的方法簽名
   - 與其他模組的呼叫關係
   - 設定欄位、事件、回呼名稱
2. 在 skill 結尾的「最近修改備忘」或「歷史紀錄」段落（若該物件有），加上一行簡述（可選）。
3. 若新增 / 刪除檔案，**同時更新 `docs/skills/README.md` 的索引表**。

---

## 2. 何時要新增 / 拆分 skill

| 狀況 | 處理方式 |
|------|----------|
| 新增一個全新類別、模組或檔案 | 在對應資料夾建立 `xxx.skill.md`；於 README 索引補上一行 |
| 既有 skill 超過 ~250 行或描述多個職責 | 拆出模組資料夾，例如 `broker/realtime/` |
| 新增完全獨立的功能領域 | 在 `docs/skills/` 底下新增資料夾 |
| 物件被移除 / 改名 | 同步刪除 / 重命名 skill；維持索引乾淨 |

## 3. Skill 檔撰寫格式建議

每份 `*.skill.md` 建議包含下列段落（可依模組性質取捨）：

1. **Skill：<物件名稱>** — 開頭一句話定位該物件做什麼。
2. **檔案位置** — 對應原始碼相對路徑。
3. **主要職責** — 條列該物件負責的事情（≤ 6 條）。
4. **主要類別 / 函式** — 重點 API；列出簽名與用途。
5. **與其他模組關係** — 引用了誰、被誰呼叫、broadcast 哪些事件。
6. **輸入 / 輸出 / Side Effect** — 寫檔、執行緒、UI 變更等。
7. **設定 / 常數** — 牽涉 `config.json` / `broker_settings.json` / 環境變數的欄位。
8. **常見錯誤 / 注意事項** — 容易踩雷的點。
9. （可選）**測試對應位置** — `build_win/src/tests/` 下對應的測試檔。

---

## 4. 不要做的事
- ❌ 直接改程式碼但不更新對應 skill。
- ❌ 在沒有先讀 skill 的情況下重構或刪除任何「看似沒被用到」的東西（多半被 GUI 或測試引用）。
- ❌ 新建巨大的單一 skill 檔包多個物件（請拆模組）。
- ❌ 把臨時筆記、進度紀錄、TODO 清單寫進 skill；skill 應描述「目前的真實樣貌」。

---

## 5. 與既有 docs 文件的關係
- `docs/` 內既有的 `*.md` 為「需求 / 評估 / 計畫」類設計文件，**屬於『為什麼這樣做』**。
- `docs/skills/` 則是「物件目前怎麼做」**的索引與摘要**。
- 兩者不衝突：寫設計或需求時更新 `docs/*.md`；完成實作時更新 `docs/skills/*.skill.md`。

- [ ] 是否改變了對外 API 或事件名稱？若是，已更新所有相關 skill？
- [ ] 是否新增 / 刪除檔案？若是，已更新 `README.md` 索引？
- [ ] skill 中描述的「與其他模組關係」仍然正確？
