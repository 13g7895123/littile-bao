## 一、背景

今日 `2026-06-04` 的實際執行資料顯示，系統已不是單純的策略邏輯問題，而是出現兩個運行層級異常：

1. `09:20` 左右第一輪策略運行期間，WebSocket 出現 timeout / connection closed。
2. `10:02:56` 第一輪引擎停止，`10:05:02` 第二輪引擎重新啟動後，`1904 正隆` 的委託/成交 log 出現雙寫，但 audit 只有一筆實際委託與成交。

這代表目前系統同時存在：

- 連線中斷後的恢復流程不夠完整
- 策略重啟後，券商 callback 可能殘留舊訂閱者，導致同一筆事件被重複處理

## 三、問題定義

本次要修的不是單一 bug，而是兩個彼此關聯的運行問題。

### 問題 A：Socket 斷線恢復流程不完整

期望行為：

- 一旦行情 WebSocket 中斷
- GUI 立即收到事件
- 立即停止當前策略
- 跳出提示視窗
- 自動重啟策略

現況：

- 雖然程式碼已經補了 `disconnect callback`
- 但今天的執行紀錄沒有看到這條新流程實際發生
- 因此要確認：今天執行的是不是舊 build，或新流程仍有覆蓋不到的斷線型別

### 問題 B：重啟後 broker callback 重複註冊

期望行為：

- 每輪 engine 啟動，只應有一組有效的 `on_order / on_filled` 訂閱
- engine 停止後，舊 callback 應解除

現況：

- `BrokerAdapter.on_filled()` / `on_order()` 只會 append callback
- `TradingEngine.stop()` 沒有解除訂閱
- 因此只要同一個 broker 實例被重用，重啟一次就可能多一組 callback

## 四、根因假設

### 4.1 Socket 斷線問題的可能根因

1. GUI 目前只接 `websocket disconnect`，但今天第一輪主要是先出現 `timeout / connection closed / underlying connection closed`
2. 斷線事件可能發生在 SDK 底層，不一定都走到我們掛的 `disconnect` handler
3. 今日 Windows `dist` 可能仍在跑舊 build，尚未包含最新自動恢復邏輯

### 4.2 重複回報問題的直接根因

1. `BrokerAdapter` 只提供 `on_order()` / `on_filled()`，沒有 `off_order()` / `off_filled()`
2. `TradingEngine.start()` 每次重啟都重新註冊一次
3. `TradingEngine.stop()` 沒有清掉舊訂閱
4. 同一個 broker 實例被保留時，第二輪會疊加 callback

---

## 五、修復目標

### 5.1 斷線恢復目標

- 對「真實 disconnect」
- 對「底層 connection closed / timeout」
- 對「start 前發現 SDK connection dead」

三種情境都能有一致恢復行為：

- log 可見
- GUI 可感知
- 策略可停
- 使用者有提示
- 可自動重啟

### 5.2 事件訂閱目標

- 同一 broker 實例在任一時間只保留一組有效 engine callback
- 重啟後 log / 委託 / 成交 / 統計不得雙寫

### 5.3 驗證目標

- 盤中重啟後，`今日第 N 檔` 計數應符合單一有效 engine
- `audit` 與 `program.log` 的委託/成交筆數一致

## 七、修復方案

## 7.1 補齊 broker callback 解除機制

在 `BrokerAdapter` 增加：

- `off_order(callback)`
- `off_filled(callback)`
- 或提供 `clear_order_subs()` / `clear_fill_subs()`

建議優先採「精準解除單一 callback」：

- 比 `clear_*` 安全
- 不會誤清 GUI 自己想保留的其他訂閱者

## 7.2 TradingEngine 明確管理訂閱生命週期

在 `TradingEngine`：

- `start()` 註冊 broker callbacks
- `stop()` 必須對應解除
- 若 `start()` 被重複呼叫，也要防重複註冊

建議做法：

- 用布林旗標 `self._broker_callbacks_registered`
- 將實際註冊 / 解除封裝成 `_attach_broker_callbacks()` / `_detach_broker_callbacks()`

## 7.3 擴大斷線事件的上拋範圍

目前 `disconnect callback` 只覆蓋 websocket `disconnect`。

應補一層「連線死亡事件」統一出口，例如：

- `_notify_disconnect("...")`
- `_notify_disconnect("underlying connection is closed")`
- `_notify_disconnect("websocket timeout")`

也就是：

- 不只在 `on("disconnect")` 觸發
- 在 `error` 或 `_is_connection_dead_error()` 命中時，也可把事件上拋到 GUI

## 7.4 GUI 端重啟流程改成可驗證狀態機

GUI 目前已有 `_socket_recovering`。

建議再補：

- 最近一次斷線時間
- 最近一次斷線原因
- 最近一次自動重啟結果

這樣之後看 log 就能回答：

- 有沒有收到斷線
- 有沒有真的停策略
- 有沒有真的成功重啟

## 7.5 補 build / runtime 驗證

今天的 repo 程式碼與 `dist` 執行結果不一致的可能性很高。

因此修完後要特別做：

- 確認 `dist` 已重新 build
- 確認 Windows 實際執行的是新版本
- 確認 log 中能看到新加的提示文字

## 九、實作步驟

### 步驟 1：先修 callback 重複註冊

原因：

- 這是今天已明確重現的 bug
- 也最容易以單元測試穩定驗證

工作項目：

1. `BrokerAdapter` 新增解除註冊 API
2. `TradingEngine.stop()` 補解除邏輯
3. `TradingEngine.start()` 補防重複註冊保護

### 步驟 2：補齊斷線死亡事件上拋

工作項目：

1. `realtime.py` 把 `disconnect`
2. `error`
3. `underlying connection is closed`

統一導向同一個 disconnect/recover callback

### 步驟 3：強化 GUI 重啟觀測性

工作項目：

1. log 明確寫出：
   - 收到斷線
   - 停止策略
   - 顯示視窗
   - 準備重啟
   - 重啟成功 / 失敗
2. 若重啟失敗，不能靜默吞掉

### 步驟 4：重 build 後做 Windows 實跑驗證

工作項目：

1. rebuild `dist`
2. 模擬或等待實際重啟情境
3. 對照 `program.log` / `audit` / recording

## 十二、風險與對策

### 風險 1：解除 callback 影響其他訂閱者

對策：

- 不用 `clear_all`
- 改用「只移除自己註冊的 callback」

### 風險 2：同一斷線事件重複上拋

對策：

- `GUI _socket_recovering` 保留
- `realtime.py` 端可再加 debounce 或 once flag

### 風險 3：重啟後沿用舊 broker 狀態不乾淨

對策：

- 在重啟前明確 `engine.stop()`
- 需要時強制重建 realtime feed
- 若必要，重新登入 broker

### 風險 4：修完 repo 但 `dist` 沒更新

對策：

- 修完後強制 rebuild
- 驗證 log 中有新版本特徵字串
