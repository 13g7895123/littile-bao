## 量測方式

系統在每次收到 tick / book 後，會記錄以下時間：

- `market_event_time`：市場事件時間戳
- `recv_time`：本機實際收到該事件的時間
- `decision_time`：本機完成策略判斷的時間

本次主要觀察指標：

- `market_to_recv_ms = recv_time - market_event_time`
- `recv_to_decision_ms = decision_time - recv_time`

若 `market_to_recv_ms` 長時間穩定約 `3000ms`，表示問題主要出在「資料到達本機之前」；若 `recv_to_decision_ms` 很小，代表本機策略程式不是主要瓶頸。

## 與本機策略程式的關係

從現有紀錄判斷，本機程式不是主要瓶頸，原因如下：

1. `recv_to_decision_ms` 幾乎都接近 `0ms`
2. 延遲在大量股票上同步出現，不像單一 symbol 邏輯卡住
3. 延遲從訂閱完成後很快就開始，且數值非常穩定
4. `market_to_recv_ms` 的穩定程度更像上游時間戳與推送到站時間之間固定落差

因此目前較合理的懷疑方向是：

- 富邦行情來源的事件時間戳本身就早於實際推送時間約 `3s`
- websocket 推送鏈路有固定排隊 / 緩衝
- SDK 對外回報的 `time` 代表的是較早的市場時間，而不是實際送出時間

---

## 對策略造成的實際影響

### 鎖板 / 撬板判斷延後

策略是依據即時 tick / book 判斷是否鎖漲停、是否撬開、是否爆量。
若資料本身落後約 `3` 秒，則：

- 買點會晚約 `3` 秒
- 賣點也會晚約 `3` 秒
- `1 秒量` 相關條件在實際市場上已經是過去資料

### 盤中異常案例容易被放大

例如 `3041 揚智` 在 `2026-06-15 10:29:23` 的進出場異常，雖然最後定位為 dry-run 下單 race condition，但其觸發基礎資料本身也已經是延後約 `3` 秒到站，對盤中策略穩定性仍有明顯影響。
