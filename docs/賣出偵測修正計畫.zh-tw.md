## 一、背景與問題定義

目前策略出場有兩條主要規則：

- `F4`：漲停板打開就賣
- `F5`：1 秒成交量爆量就賣

使用者這次提出的核心疑問不是「有沒有賣掉」，而是更精確的兩件事：

1. 若某檔股票在仍屬漲停狀態下，先出現 `1 秒量 >= 499` 或 `> 499` 的爆量訊號，是否應該優先以 `F5` 立即賣出？
2. 若原始行情顯示 `F5` 訊號早於 `F4`，為何最終交易紀錄仍顯示為 `F4`？

這已不是單純的門檻問題，而是「**條件成立的真實先後順序**」與「**策略迴圈當下採用哪個理由送單**」之間存在落差。

本計畫書的目的，是將這個落差拆解清楚，並提出完整修正方案，讓：

- 觸發門檻符合使用者語意
- 賣出理由符合原始行情先後
- 測試可重現
- 日後報表與復盤不再混淆

### 2.3 這些案例代表的真實問題

問題不只是「門檻 499 還是 500」。

更大的問題是：

- 原始市場事件中，`F5` 常常比 `F4` 更早成立
- 但現行策略不是根據「最早成立的事件時間」決定原因
- 而是根據 `_tick()` 執行時的固定判斷順序，先看 `F4`、再看 `F5`

因此，交易紀錄會出現：

- 實際上是先爆量
- 但最終賣出原因被記成「漲停板打開」

這會直接影響：

- 盤後分析
- 策略優化方向
- 使用者對策略是否照規則行動的信任

## 三、現行邏輯分析

### 3.1 現行出場邏輯位置

出場邏輯位於 [build_win/src/engine.py](/home/jarvis/project/bonus/03_littile-bao/build_win/src/engine.py:655) 附近。

現況順序如下：

1. 先判斷 `F4`
2. 若 `F4` 沒命中，才判斷 `F5`
3. 一旦 `reason` 已被 `F4` 設定，`F5` 不再覆蓋

### 3.2 現行 `F4` 定義

`F4` 本質上不是看「何時首次開板」，而是看 `_tick()` 執行當下：

- `open_ticks >= f4_open_ticks_to_sell`
- 且 `state.last_price is not None`

也就是說，當 `_tick()` 跑到這一輪時，只要此刻板已開，`F4` 就可能先佔掉 `reason`。

### 3.3 現行 `F5` 定義

`F5` 是：

- `state.last_1s_vol > cfg.volume_spike_sell_threshold`

預設門檻目前為 `499`，但運算子是 `>`，因此實際語意是：

- `500` 以上才觸發
- `499` 不觸發

### 3.4 現行設計的盲點

現行設計只有「本輪 `_tick()` 時，哪個條件先被寫入 `reason`」。

它沒有保留：

- `F4` 首次成立時間
- `F5` 首次成立時間
- 本次賣出到底是哪個條件先出現

換言之，系統現在只有「輪詢時點結果」，沒有「事件先後歷史」。

## 四、根因整理

### 根因 1：`F4` 與 `F5` 採固定判斷順序，不採最早事件時間

這是四筆誤標成 `F4` 的直接原因。

### 根因 2：`F5` 門檻採用 `> 499`，與使用者語意「499 就算」不一致

這是 `9136` 爭議的直接原因。

### 根因 3：引擎主循環是秒級 `_tick()`，但錄製行情是毫秒級事件

即使同一秒內 `F5` 先發生、`F4` 後發生，只要 `_tick()` 執行時兩者都已成立，現行結果仍會偏向 `F4`。

### 根因 4：目前缺少專門的單元測試驗證「同秒雙條件競合」

因此這個問題即使存在，也不會在 CI 或本地測試被攔下。

## 七、需求規格

### 7.1 門檻規格

需先明確定義 `F5` 的正式語意。

建議改為：

- `last_1s_vol >= volume_spike_sell_threshold`

若保留預設門檻 `499`，則語意變成：

- `499` 就觸發

這與使用者目前口語規則一致。

### 7.2 競合規格

當 `F4` 與 `F5` 都成立時，不應再以程式判斷順序決定，而應改為：

1. 比較 `F4` 首次成立時間
2. 比較 `F5` 首次成立時間
3. 以較早者決定本次賣出原因

### 7.3 並列情境規格

若極端情況下 `F4` 與 `F5` 的首次成立時間完全相同，需定義 tie-break 規則。

建議：

- 同時間戳時優先 `F5`

理由：

- 使用者目前主要關注的是「爆量先抓到就立即賣」
- `F5` 本質上是更偏向主動風險訊號

### 7.4 單次送單規格

無論原因是 `F4` 或 `F5`，都必須維持現有保護：

- 若 `state.pending = True`，不得重複送出賣單

### 7.5 紀錄規格

交易紀錄、策略觸發紀錄、log 至少應能保留：

- 本次採用的出場策略 `F4/F5`
- 觸發細節
- 若另一策略也成立，可選擇附帶記錄競合資訊

建議新增可選的附帶資訊：

- `first_f4_time`
- `first_f5_time`
- `winner_strategy`

## 八、設計方案

### 8.1 方案原則

不建議只把 `if F5 ...` 移到 `if F4 ...` 前面。

原因：

- 這只能改變「同一輪 `_tick()` 的優先順序」
- 但無法保證真實市場事件的先後
- 也無法保留盤後分析所需的證據

因此本次建議採「**事件時間驅動 + `_tick()` 執行時決策**」設計。

### 8.2 建議資料結構擴充

在 `StockState` 增加以下欄位：

```python
f4_first_trigger_at: Optional[float]
f5_first_trigger_at: Optional[float]
f4_trigger_snapshot: Optional[dict]
f5_trigger_snapshot: Optional[dict]
```

用途：

- 記錄本輪尚未送單前，`F4/F5` 首次成立的時間
- 保留當時的關鍵值，方便決策與除錯

### 8.3 訊號記錄時機

#### `F4`

當系統判斷出：

- `open_ticks >= threshold`

且 `f4_first_trigger_at` 尚未寫入時，立即記錄首次觸發時間。

#### `F5`

當系統判斷出：

- `last_1s_vol >= threshold` 或 `> threshold`（依最終需求）

且 `f5_first_trigger_at` 尚未寫入時，立即記錄首次觸發時間。

### 8.4 賣出決策方式

當 `_tick()` 進入出場段且尚未 `pending`：

1. 先更新 `F4/F5` 是否新成立
2. 檢查兩者是否至少一者已成立
3. 若只有一者成立，採用該策略
4. 若兩者都成立，比較 `f4_first_trigger_at` 與 `f5_first_trigger_at`
5. 以較早者作為最終 `reason` 與 `strategy`

### 8.5 送單後清理

一旦 `_do_sell()` 送單成功，需清理本次競合狀態，避免舊資料污染後續判斷。

建議清掉：

- `f4_first_trigger_at`
- `f5_first_trigger_at`
- `f4_trigger_snapshot`
- `f5_trigger_snapshot`

### 8.6 是否要在 `_on_book()` / `_on_tick()` 當下即送單

本次不建議直接改成事件 callback 當下送單，理由：

- 會擴大本次變更範圍
- 會碰到更多執行緒與重入問題
- 現有架構主要仍以 `_tick()` 為策略決策入口

因此本次設計以「先記錄最早成立時間，再由 `_tick()` 做最終送單」為主，是較穩健的過渡方案。

## 十三、風險與對策

| 風險 | 說明 | 對策 |
|---|---|---|
| 行為改變導致歷史結果不同 | 交易整理將與舊版不一致 | 文件中明示「修正前/修正後」差異 |
| `499` 改 inclusive 後觸發數量增加 | 部分股票會更早賣出 | 補回測/錄製驗證，先以 dry-run 驗證 |
