## 二、逐題分析

### 問題 1：100 張以內買到，但板上還有 50 張賣，4 點會不會誤賣？

> 你的場景：「100 張內買到，可是還剩 50 張掛賣，4 點時委賣還剩 50 張，會不會又因為功能 4 賣掉？」

**結論：不會。** ✅

原因看 [engine.py 的 `_on_book`](build_win/src/engine.py)：
```python
if state.ask0_price == limit_up:
    state.ask_qty_at_limit = state.ask0_volume   # ← 50 張
    state.is_at_limit_up = True                  # ← 仍是 True
else:
    state.is_at_limit_up = False                 # ← 板被打開才會 False
```

功能 4 只看 `is_at_limit_up`：
```python
if cfg.f4_enabled and not state.is_at_limit_up:
    reason = "委買漲停，市場打開，市價出場"
```

只要 ask\[0]（最低委賣價）**還是漲停價**，不論還剩 100 張還是 1 張，`is_at_limit_up` 都仍是 True，**功能 4 不會觸發**。
功能 4 觸發條件是「最佳賣價不再等於漲停價」＝有人把漲停板上的賣單全部吃光、最佳賣價跳到漲停價之下。

### 問題 2：功能 4「能不能設定打開幾檔才賣」

> 你的訴求：「怕被洗掉，能不能設定打開幾檔位才賣？」

**結論：目前不行，建議新增；解法應以「檔位」為主，不用時間當必要條件。** ❌

目前 [_tick()](build_win/src/engine.py) 是 binary：板一打開就賣，沒有容忍度。
洗盤情境真實會發生：板被吃開一兩檔再迅速回補，瞬間打開那一刻就被市價殺出。

這個問題可以用五檔資料處理。系統已經訂閱 `books`，`_on_book()` 也會維護：

```python
state.ask0_price    # 委賣一檔價格
state.ask0_volume   # 委賣一檔張數
state.bid0_price    # 委買一檔價格
state.bid0_volume   # 委買一檔張數
```

所以可以把「打開幾檔才賣」定義成：**最佳委賣價 `ask0_price` 距離漲停價低了幾個 tick**。

例如：

| 設定 | 行為 |
|---:|---|
| `1` | 打開 1 檔就賣，等同目前「一打開就賣」 |
| `2` | 只打開 1 檔先忍住，打開到第 2 檔才賣 |
| `3` | 打開 1～2 檔先忍住，打開到第 3 檔才賣 |

#### 建議新增設定

```python
# config.py
f4_open_ticks_to_sell: int = 1   # 打開幾檔才賣；1 = 打開 1 檔就賣
```

engine 改寫：
```python
limit_up = Decimal(str(info.limit_up))
tick = tick_size(limit_up)
open_ticks = 0
if state.ask0_price is not None and state.ask0_price < limit_up:
    open_ticks = int((limit_up - state.ask0_price) / tick)

if open_ticks >= cfg.f4_open_ticks_to_sell:
    reason = f"漲停板打開 {open_ticks} 檔，達出場門檻"
```

這樣可避免「只吃開 1 檔就被洗掉」造成的誤殺。若未來還要防「一瞬間打到第 N 檔又秒封回」，可以再加秒數條件，但本次主方案先以**檔位**為準，因為五檔資料已足夠支援這個設定。

### 問題 3：「開盤就漲停」獨立開關

> 你的訴求：「開盤就漲停的股票，能不能做個開關，看要不要排隊？」

**結論：目前不行，建議新增。** 🟡

目前 `f12_enabled` 控制的是**封鎖規則**（開盤即漲停 + 已賣過 → 封鎖），但**沒有專門開關決定要不要追開盤就漲停的股票**。

[engine.py 進場邏輯](build_win/src/engine.py)目前完全沒有針對 `info.open_limit_up` 做進場端的特殊處理：開盤就漲停的股票照樣可以進場（只要符合其他條件）。

#### 建議新增

```python
# config.py
f_open_limitup_entry_enabled: bool = True   # 是否允許進場開盤就漲停的股票
```

engine：
```python
# 進場前
if not cfg.f_open_limitup_entry_enabled and info.open_limit_up:
    return    # 開盤即漲停 → 不追
```

#### 加入項目

| 項目 | 說明 |
|---|---|
| 設定欄位 | `f_open_limitup_entry_enabled` |
| GUI 控制 | 新增 checkbox：「允許追開盤即漲停」 |
| 預設值 | `True`，維持目前行為 |
| 關閉時 | `info.open_limit_up == True` 的股票直接跳過進場 |
| 與 F12 差異 | F12 是「開盤即漲停 + 已賣過」才封鎖；此項目是不論是否賣過，都可選擇不追開盤即漲停 |

### 問題 4：「賣掉後不追」會不會卡到隔天？

> 你的問句：「如果今天我賣了，明天它又準備漲停、也符合第二根，是不是又不會買了？」

**結論：⚠️ 取決於程式有沒有跨日重啟，目前邏輯有風險。**

#### 4-1 看程式碼

[engine.py 的 `StockState`](build_win/src/engine.py)：
```python
self.entry_blocked: bool = False   # 取消委託 / 賣出後 = True
self.sold_today: bool = False      # 賣出後 = True
```

這兩個欄位**只在引擎物件存活期間有效**。每次 GUI 按下「啟動」，[gui.py._start_trading()](build_win/src/gui.py) 會 `TradingEngine(...)` 重新建立 → `_states` 全部重置，`entry_blocked` / `sold_today` 也歸零。

#### 4-2 風險情境

| 場景 | 結果 | 風險 |
|---|---|---|
| 收盤後關閉程式，隔天早上重新啟動 | 全部重置，第二天可正常追 | ✅ 安全 |
| 程式 24 小時不關 / 跨日 | `sold_today`、`entry_blocked` 仍為 True，隔天**不會買** | ⚠️ 卡住 |
| 中午重啟 | 同樣會重置，但下午盤剩下的時間能否來得及又是另一回事 | 🟡 |

#### 4-3 建議補強

在 engine 加入或保留「跨日 reset」：
```python
self._trading_date = date.today()

def _tick(self, ...):
    today = date.today()
    if today != self._trading_date:
        for st in self._states.values():
            st.entry_blocked = False
            st.sold_today = False
            st.candle_index = 0
            st.position_qty = 0
            st.limit_up_since = None
        self._daily_trade_count = 0
        self._trading_date = today
```

或更簡單：偵測時間到 `08:50` 自動重置一次。

#### 4-4 加入項目：一鍵賣出所有本系統買入股票

除了跨日 reset，建議再新增一個明確的人工風控動作：「賣出所有本系統買入股票」。

這個功能建議限定為**本策略引擎當日買入、且目前仍由 engine 記錄為持倉的股票**，不要直接賣出整個帳戶庫存，避免誤賣使用者手動買進或隔日留倉的部位。

| 項目 | 說明 |
|---|---|
| GUI 動作 | 新增按鈕：「全部策略持股賣出」 |
| engine 方法 | `sell_all_strategy_positions(reason="manual_sell_all")` |
| 賣出範圍 | `state.position_qty > 0` 且由本策略買入的股票 |
| 下單方式 | 逐檔送出 SELL 委託，價格可沿用現有出場下單策略 |
| 風控提醒 | 點擊前跳確認視窗，顯示將賣出的代碼與張數 |
| 不處理範圍 | 帳戶中非本策略買入、或 engine 沒有接管的隔日庫存 |

這個按鈕用途是手動收斂風險：例如策略邏輯還在驗證、行情異常、或盤中想立刻把本策略買到的部位全部出清時，不需要逐檔手動取消/賣出。

### 問題 5：功能 13 與功能 1 互斥開關

> 你寫的 13 條：「委賣在漲停還在消化量，成交幾張我就要進場（可關掉功能 1）」

**結論：⚠️ 你這條跟我目前實作的「功能 13」其實是兩件事，且尚未實作。** ❌

| 你描述的 13 | 我現有的 `f13_enabled` |
|---|---|
| 漲停消化量達 N 張 → 進場 | 當天最多成交 5 檔 → 達上限不再進場 |

#### 建議

把現有 `f13_daily_max_trades` 改名或保留，**新增**一條「漲停消化量進場」：
```python
# config.py
f_consume_enabled: bool = False        # 是否啟用「消化量達 N 張就進場」
consume_qty_threshold: int = 499       # 累計成交（漲停價）達 N 張即進場
consume_mutex_with_f1: bool = True     # 與功能 1 互斥，避免條件方向衝突
```

注意它和功能 1（10 點前 + 委賣 < 100 張才進場）邏輯衝突：
- 功能 1：**委賣張數越少越好**（要快漲停）
- 此新功能：**消化量越多越好**（已經有人吃進）

GUI 應提示使用者「兩者擇一啟用」，或在 engine 內檢查同時 enabled 時擇優先。

#### 加入項目

| 項目 | 說明 |
|---|---|
| 設定欄位 | `f_consume_enabled`、`consume_qty_threshold`、`consume_mutex_with_f1` |
| GUI 控制 | 新增「漲停消化量進場」區塊，可輸入 N 張 |
| 資料來源 | `trades`，只累計成交價等於漲停價的成交量 |
| 狀態欄位 | `state.limit_up_consumed_qty`，每日或離開漲停狀態時依規則重置 |
| 進場條件 | `state.limit_up_consumed_qty >= consume_qty_threshold` |
| 互斥規則 | 若 `consume_mutex_with_f1=True`，GUI 勾選此項時自動關閉 F1，或 engine 啟動時阻擋同時啟用 |

### 問題 6：隔天開盤沒漲停，碰到功能 4 會自動賣？

> 你的問句：「如果我買到的隔天，開盤沒漲停，又碰到第 4 點，那不就又自動幫我賣掉？」

**結論：⚠️ 會！這是現行邏輯最大的隱患。**

#### 6-1 現行 F4 條件

```python
if state.position_qty > 0:
    if cfg.f4_enabled and not state.is_at_limit_up:
        reason = "委買漲停，市場打開，市價出場"
        self._do_sell(...)
```

判斷只看「**目前不是漲停**」＝任何時刻只要持倉、且當下不是漲停板，就賣。

#### 6-2 觸發條件

要讓這個 bug 出現，需要同時滿足：
1. **程式跨日不重啟**（否則 `position_qty` 重置為 0）
2. **隔天 broker 帳戶輪詢沒回填部位到 engine 的 StockState**（目前確實沒有把帳戶持倉同步進 engine._states，只在 GUI 持倉表）
3. 隔天該股一開盤不是漲停板

實際上目前 [build_win/src/broker/account.py](build_win/src/broker/account.py) 抓的庫存只進 GUI 持倉表，**沒有進 engine 的 `state.position_qty`**，所以 engine 在跨日後其實會把所有 `position_qty` 維持為 0（沒有同步補回），F4 不會誤觸發。

但這也代表另一個問題：**昨天買今天放的庫存，engine 不知道有部位 → 今天若該股板被打開也不會自動賣出**。等於跨日庫存目前是「啞的」。

#### 6-3 建議

加上「F4 必須先到過漲停才生效」防護：

```python
if state.position_qty > 0:
    has_been_at_limit = state.candle_index > 0 or state.entry_price is not None
    if (cfg.f4_enabled
            and has_been_at_limit              # ← 必須當日曾到過漲停
            and not state.is_at_limit_up
            and state.last_price is not None):
        reason = "委買漲停，市場打開，市價出場"
```

並且在 `_do_sell` 內也檢查：「若 entry_price 是昨日的、不是今日的，就不執行 F4 自動賣」。

最徹底的做法：
- 引擎啟動時 **不接管隔日庫存**（隔日庫存只在 GUI 顯示，不參與策略），
- 或者另設「隔日留倉處理」開關。

#### 加入項目

| 項目 | 說明 |
|---|---|
| 設定欄位 | `f4_require_today_limitup` |
| 預設值 | `True` |
| 防護條件 | F4 出場前必須確認該股「當日曾經到過漲停」 |
| 建議狀態欄位 | `state.touched_limit_up_today` 或沿用 `state.candle_index > 0` |
| 隔日庫存原則 | 預設不由 F4 自動賣出隔日庫存，除非未來另開「接管隔日庫存」設定 |
| 搭配問題 4 | 一鍵賣出功能只賣本策略買入部位，避免用 F4 誤處理隔日庫存 |
