# 盤中行情錄製與復盤功能評估

> 文件狀態：**評估階段**（尚未實作）
> 撰寫日期：2026-05-11
> 目的：評估在現有專案中加入「盤中 Socket 行情錄製 → 收盤後復盤」能力的可行性、架構規劃、風險與工作量估算。

---

## 1. 需求摘要

使用者情境：
- 當天策略啟用 10~20 元監控區間
- 啟用時間 11:00 ~ 11:30
- 收盤後希望能重現該時段的盤中情境，用以：
  1. 確認**即時監控**邏輯是否正確（價格、漲跌、1 秒量、漲停委賣張數等顯示）
  2. 確認**策略執行**是否按預期觸發（準備進場 / 進場 / 出場 / 條件不符）
- 復盤模式**禁止**任何正式下單

## 2. 現有架構摘要（與本需求相關）

### 2.1 即時行情資料流

```
富邦 SDK (_fubon_neo.pyd, WebSocket)
        │  推送 JSON 字串
        ▼
broker.realtime.FubonRealtimeFeed._on_raw_message(msg)
        │  解析 event / channel
        ▼
        ├─ _dispatch("trades", data) → _emit_tick(TickEvent)
        └─ _dispatch("books",  data) → _emit_book(BookEvent)
                                              │
                                              ▼
                                  engine.TradingEngine._on_tick / _on_book
                                              │
                                              ▼
                                  狀態更新 / 策略判斷 / get_summary()
                                              │
                                              ▼
                                  gui._render_monitor (1Hz 輪詢)
```

### 2.2 關鍵注入點

| 注入點 | 檔案 | 說明 |
|--------|------|------|
| 原始 JSON 進入點 | `broker/realtime.py::FubonRealtimeFeed._on_raw_message` | 最完整、未經修飾的訊息（適合**錄製**） |
| 解析後事件 | `_emit_tick` / `_emit_book` | 已轉成 `TickEvent` / `BookEvent` DTO（適合**輕量錄製**） |
| 引擎輸入 | `TradingEngine._on_tick` / `_on_book` | callback 由 `feed.on_tick(...)` 註冊 |
| Feed 工廠 | `broker.adapter.FubonAdapter.create_realtime_feed()` | 啟動策略時建立 feed |

### 2.3 與復盤直接相關的特性

- `RealtimeFeed` 是抽象基類，已有 `MockRealtimeFeed`（隨機產 tick/book）
- → 可新增 `ReplayRealtimeFeed`（讀檔回放），完全相容現有 callback 介面
- `TradingEngine` 不需任何改動即可接受 Mock 或 Replay feed

## 3. 設計方案

### 3.1 整體架構

```
┌──────────── 盤中：錄製 ────────────┐    ┌──────────── 收盤後：復盤 ────────────┐
│                                     │    │                                       │
│ FubonRealtimeFeed._on_raw_message   │    │ ReplayRealtimeFeed                    │
│  ├─ 原行為（emit tick/book）        │    │  ├─ 讀取 .ndjson.gz 行情檔            │
│  └─ NEW: 寫入 RecordingWriter ──┐   │    │  ├─ 依時間戳重播 tick / book          │
│                                  │   │    │  ├─ 可調倍速 / 區間 / 暫停            │
│                                  ▼   │    │  └─ 注入到 TradingEngine 同樣 callback│
│                          record/      │    │                                       │
│                          YYYYMMDD/    │    │ TradingEngine（同一份 code）          │
│                          session_*.ndjson.gz                                       │
│                          + meta.json  │    │ OrderManager → 強制 DryRunOrderManager│
└─────────────────────────────────────┘    └───────────────────────────────────────┘
```

### 3.2 錄製檔格式

#### 3.2.1 目錄結構

```
log/
  recordings/
    20260511/
      session_110000_113000.meta.json    # 工作階段中繼資料
      session_110000_113000.ticks.ndjson.gz   # 行情逐筆
      session_110000_113000.events.ndjson.gz  # 策略事件（選擇性）
```

#### 3.2.2 `meta.json` 內容範例

```json
{
  "session_id": "20260511_110000_113000",
  "start_ts":   "2026-05-11T11:00:00+08:00",
  "end_ts":     "2026-05-11T11:30:00+08:00",
  "config_snapshot": { "...": "TradingConfig 完整 asdict()" },
  "symbol_universe": [
    { "code": "1234", "name": "...", "prev_close": 15.0, "limit_up": 16.5, "market": "TSE" }
  ],
  "stats": {
    "tick_count": 12345,
    "book_count": 8765,
    "raw_message_count": 21123,
    "missed_count": 0
  },
  "app_version": "2.x.x",
  "sdk_version": "fubon_neo-2.2.8"
}
```

#### 3.2.3 `ticks.ndjson.gz`（每行一筆 JSON）

採用「原始 JSON + 接收時間戳」的雙軌格式：

```json
{"t": 1715388000.123, "kind": "raw", "msg": "<原始 SDK 訊息字串>"}
{"t": 1715388000.130, "kind": "tick", "code": "2330", "price": 600.0, "volume": 5, "ts": "..."}
{"t": 1715388000.150, "kind": "book", "code": "2330", "bids": [...], "asks": [...]}
```

- `t` 為**本機接收時間**（單調遞增），復盤排序基準
- `raw` 行保留原始 JSON 字串，可作為後續解析改動時的回測依據
- `tick` / `book` 行為已解析 DTO，復盤時可跳過 `raw` 直接走快速路徑

#### 3.2.4 預估儲存量

- 單筆 tick JSON ≈ 200 bytes（壓縮後 ≈ 40 bytes）
- 半小時 × 200 檔 × 平均 2 ticks/sec ≈ 720,000 筆
- → **壓縮後約 25~30 MB / 30 分鐘**（可接受）

## 4. 風險與限制

### 4.1 技術風險

| 風險 | 等級 | 緩解措施 |
|------|------|----------|
| 原始 JSON 字串內容若含敏感資訊（憑證、token） | 低 | 上線前抽樣檢視，必要時加白名單過濾 |
| 高頻時段（開盤 9:00~9:05）寫入 IO 阻塞 | 中 | 使用獨立 writer thread + queue（如 `queue.Queue(maxsize=10000)`），溢位則丟棄 raw 行（保留 tick/book） |
| 復盤時點數據與真實 GUI 渲染速度不同步 | 低 | `ReplayRealtimeFeed` 用 `time.sleep(dt / speed)` 控速 |
| 復盤過程中誤觸發實單 | **高** | 啟動復盤時，強制把 broker 換成 `MockAdapter` 或把 OrderManager 換成 `DryRunOrderManager`，並在 UI 上鎖死實單切換 |
| 跨日策略狀態（庫存、已實現損益）污染 | 中 | 復盤啟動時建立全新 `TradingEngine` 實體，與正式線完全隔離 |

### 4.2 邏輯限制

1. **盤後查詢類 API 無法重播**：例如 `previous_trading_days_api`、`snapshot`。
   → 復盤時需從 `meta.json` 的 `symbol_universe` 還原 `SymbolInfo`，不再打 API。

2. **訂閱清單固定**：錄製時訂閱了哪些檔，復盤只能回放那些。
   - 若使用者復盤時調整價格區間到 30~40 元（原本錄製 10~20 元），結果**仍只能看到 10~20 元當時訂閱的標的**。
   - 文件需明確標註此限制，避免使用者誤判為 bug。

3. **委託 / 成交回報無法重播**：實單 callback 是 broker 推送，非 WebSocket 行情。
   → 復盤模式下使用 `DryRunOrderManager` 自我產生模擬 fill。

4. **時鐘相關策略**：例如「9:30 後才允許進場」這類用 `datetime.now()` 的判斷會以「真實當下時間」評估，而非錄製當時時間。
   → 需在 engine 增加 `clock` 抽象層，復盤時注入 `ReplayClock`。**這是最大改動點**。

### 4.3 對既有功能的衝擊

- **錄製功能**：只在 `_on_raw_message` 加一行 `if self._recorder: ...`，零功能衝擊
- **復盤功能**：需新增「clock 抽象層」改動 `TradingEngine` 內所有 `datetime.now()` / `time.time()` 呼叫 → 約 10~20 處
- 既有單元測試需檢查是否有寫死 `datetime.now()` 的測試

## 7. 待使用者決策事項

1. **錄製範圍**：要錄「所有訂閱檔」還是「僅符合 F9 區間檔」？
   - 建議：錄**所有訂閱檔**（含放寬 ±10% 的範圍），保留復盤時調整 F9 區間的彈性

2. **錄製啟動時機**：策略啟用時自動錄製？還是獨立開關？
   - 建議：獨立開關，預設關閉（避免不必要的磁碟壓力）

3. **保留天數**：預設保留幾天？
   - 建議：7 天，使用者可調整

4. **是否需要錄製策略事件（_on_strategy_event）**：
   - 建議：階段 1 先不錄；階段 2 加入，用於比對「同樣行情下，策略是否做出相同決策」

5. **時鐘抽象層的改動是否接受**：
   - 這是 engine 內較大的改動，需確認是否願意承擔回歸風險
   - 替代方案：復盤時不抽象時鐘，僅展示行情變化，不執行時間相關判斷（功能折衷）
