# 動態換股池（Dynamic Universe Rotation）計畫書

**版本：** 1.0  
**日期：** 2026-06-02  
**狀態：** 規劃中

---

## 一、背景與目標

### 現況

目前程式在策略啟動時（`engine.start()`）會一次性選出候選股，訂閱最多 500 檔後固定不動，盤中不再替換。若某檔股票中途不再符合條件，它仍占著訂閱名額、消耗 WebSocket 資源。

### 目標

實作「每秒評估一次、動態替換不合格股票」的機制：

1. **09:00 策略啟動**：從全市場篩出符合條件的前 500 檔，訂閱行情。
2. **每秒評估**：對清單內每一檔股票判斷是否仍符合篩選條件；不符合者移出、補入新符合者。
3. **保護規則**：有持倉 / 有委託在外 / 已有 K 線進場紀錄的股票不得被汰換。

---

## 二、篩選條件定義

### 進入條件（全部須同時滿足）

| 條件 | 說明 | 對應欄位 |
|------|------|----------|
| 漲幅介於 8%–10% | `(last_price - prev_close) / prev_close` 介於 0.08～0.10 | `state.last_price`、`state.info.prev_close` |
| 近 1 秒成交量 ≥ 500 張 | 滑動視窗 `last_1s_vol` | `state.last_1s_vol` |
| 非處置股 | `is_disposal = False` | `state.info.is_disposal` |
| 非注意股 | `is_attention = False` | `state.info.is_attention` |
| 非限當沖股 | `is_day_trade_restricted = False` | `state.info.is_day_trade_restricted` |

> **漲幅 8%–10%** 的意思：  
> - 下限 8%：股票已有明顯拉升動能，不追太早。  
> - 上限 10%：漲停價就是 10%，超過 10% 不存在（物理上限）。  
> 實際判斷：`0.08 <= (last_price / prev_close - 1) <= 0.10`

### 汰換條件（任一觸發即移出）

- 漲幅 < 8% 或 > 10%（含回落、停板）
- `last_1s_vol` < 500 張（冷卻、無人追）
- `last_price` 為 `None`（尚未收到行情）
- 例外：有持倉 / 有委託 / 已有 K 線紀錄 → 保護，不汰換

## 三、技術可行性評估

### 3.1 是否需要額外訂閱？

**結論：不需要額外付費訂閱，但需要一個「全市場監控池」。**

| 問題 | 答案 |
|------|------|
| 富邦 API 是否允許中途更換訂閱標的？ | ✅ 允許，`feed.subscribe()` 可重複呼叫；`replace_universe()` + `resubscribe_feed()` 已實作 |
| 單次最多訂閱幾檔？ | 500 檔（5 條 WebSocket × 每條 100 檔 × 2 channel） |
| 全市場代碼數量？ | 上市約 950 檔 + 上櫃約 800 檔 = 約 1750 檔 |
| 是否要同時訂閱全市場 1750 檔？ | ❌ **不能**，超過額度（500 檔上限） |
| 解決方案 | 見「補充行情機制」 |

### 3.2 行情補充機制（關鍵設計）

由於無法同時訂閱全市場，需要一個兩層結構：

```
─────────────────────────────────────────────────────
 層級 1：全市場快照（REST，每 N 秒更新）
   用途：判斷哪些股票「可能」符合條件（靜態篩選）
   資料：prev_close、quote_price（當下最新成交）
   API：sdk.marketdata.rest_client.stock.snapshot.quotes(market=...)
   限制：300 次/分鐘；全市場一次兩個 call，每 N 秒呼叫一次即可

 層級 2：即時行情 WebSocket（動態精確判斷）
   用途：已入池的 500 檔精確計算 last_1s_vol、last_price
   API：現有 trades + books 訂閱
   限制：最多 500 檔
─────────────────────────────────────────────────────
```

### 3.3 「候補池」設計

```
全市場快照（REST 輪詢，60 秒更新一次）
  └── 靜態篩選（漲幅 8-10%、排除特殊股）
        └── 候補排名（依昨量由大到小）
              └── 前 500 名 → 訂閱即時行情
                    └── 動態判斷（每秒，需 last_1s_vol ≥ 500 張）
                          └── 不合格 → 汰換 → 從候補列補入
```

## 四、詳細實作計畫

### Phase 1：新增動態換股池條件設定（`config.py`）

在 `TradingConfig` 新增以下欄位：

```python
# ── 動態換股池 ────────────────────────────────────────────
dynamic_pool_enabled: bool = False      # 是否啟用動態換股池
dynamic_pool_change_pct_min: float = 8.0   # 漲幅下限（%）
dynamic_pool_change_pct_max: float = 10.0  # 漲幅上限（%）
dynamic_pool_vol_min: int = 500            # 1 秒量門檻（張）
dynamic_pool_refresh_interval: int = 60   # 全市場快照更新間隔（秒）
dynamic_pool_max_size: int = 500          # 股票池上限（含保護標的）
```

### Phase 2：新增 `DynamicPoolManager`（新檔 `pool_manager.py`）

**主要職責：**

```python
class DynamicPoolManager:
    """
    管理動態換股池。
    - 持有全市場快照（REST 定時更新）
    - 每秒評估現有訂閱池，汰換不合格股票
    - 補入候補清單中下一個符合靜態條件的股票
    """

    def __init__(self, config, loader, feed_replace_fn, log_fn): ...

    def start(self): ...            # 啟動背景輪詢（全市場快照）
    def stop(self): ...             # 停止

    def tick(self, states: dict, now: float) -> dict:
        """
        每秒被 engine._loop() 呼叫。
        回傳 {"removed": [...], "added": [...]} 供 engine 記錄 log。
        會直接操作 engine._states（透過 engine.replace_universe + resubscribe）。
        """
    ...

    def _is_eligible(self, state) -> bool:
        """判斷某檔是否符合即時留池條件（漲幅 + 1秒量）。"""
    ...

    def _is_protected(self, state) -> bool:
        """有持倉 / 有委託 / 已進場 → 不汰換。"""
    ...

    def _refresh_candidates_from_snapshot(self): ...  # REST 呼叫（定時）
```

### Phase 3：整合進 `TradingEngine`

修改 `engine.py`：

1. `__init__`：若 `config.dynamic_pool_enabled`，建立 `DynamicPoolManager`。
2. `start`：啟動 `DynamicPoolManager`。
3. `_loop`：每秒呼叫 `pool_manager.tick(self._states, now)`，若有變動則呼叫 `replace_universe` + `resubscribe_feed`。
4. `stop`：停止 `DynamicPoolManager`。

**重要：換股時的流程**

```
1. pool_manager.tick() 計算出 removed、added
2. engine.feed.stop()
3. engine.replace_universe(new_symbol_infos)  ← 保留保護標的
4. engine.resubscribe_feed()                  ← 重新訂閱
5. log 新增/移除的股票代號
```

> ⚠️ `feed.stop()` + `resubscribe_feed()` 涉及 WebSocket 重連，**每秒執行代價極高**。  
> 必須加節流邏輯：**每次實際替換至少間隔 5 秒，且替換名額 ≥ 1 檔才執行**。

### Phase 4：GUI 設定介面

在設定面板新增「動態換股池」分組：

| 控制項 | 說明 |
|--------|------|
| 啟用動態換股池（CheckBox） | `dynamic_pool_enabled` |
| 漲幅下限（SpinBox，0.1–10%） | `dynamic_pool_change_pct_min` |
| 漲幅上限（SpinBox，0.1–10%） | `dynamic_pool_change_pct_max` |
| 1 秒量門檻（SpinBox，100–9999 張） | `dynamic_pool_vol_min` |
| 快照更新間隔（SpinBox，30–300 秒） | `dynamic_pool_refresh_interval` |

### Phase 5：日誌與狀態顯示

- 每次汰換在 Log 面板顯示：`[動態池] 移除 2330（漲幅 7.8%），補入 3481（漲幅 9.1%）`
- GUI 標題列顯示目前池中股票數量：`監控中：423 / 500`

## 五、API 額度影響分析

| 動作 | 頻率 | 額度消耗 |
|------|------|---------|
| 全市場快照（TSE + OTC） | 每 60 秒 2 次 | 2 次/分鐘（遠低於 300/min 上限） |
| WebSocket 重新訂閱 | 最多每 5 秒一次（有替換時） | 不改變總訂閱數（仍 ≤ 500 檔 × 2 channel） |
| 即時行情推送 | 與現況相同 | 不增加 |

**結論：無需升級任何訂閱方案，現有帳號額度即可支撐。**

## 六、風險與限制

| 風險 | 說明 | 緩解措施 |
|------|------|---------|
| WebSocket 重連延遲 | 每次 `resubscribe_feed()` 需 1–3 秒重建連線 | 節流至少 5 秒才替換一次；替換時保護有持倉股票 |
| REST 快照延遲 | 全市場快照最多延遲 60 秒才更新 | 候補名單可能略落後即時狀況；屬可接受誤差 |
| 同時有多檔觸及條件 | 500 檔上限下，同時有 50 檔符合但都擠在池外 | 依昨量排序，量大者優先入池 |
| 換股動作中斷行情接收 | 替換期間新加入的股票有 1–3 秒無行情 | 進場邏輯本就需等第一筆 tick，不影響安全性 |
| 全市場 REST 快照包含非普通股 | 可能混入 ETF、特別股等 | 篩選時限制代碼為 4 位純數字（現有邏輯已做） |

## 八、不需要額外訂閱功能的理由

> **明確結論：以現有富邦帳號的 API 額度，可完整實作此功能，無需升級或購買任何額外服務。**

- 動態換股池的核心操作是「重新整理現有 500 個 WebSocket 訂閱名額的分配」，總量不變。
- 全市場快照使用 REST API（每 60 秒 2 次），遠低於富邦 300 次/分鐘的限制。
- 唯一的代價是 `resubscribe_feed()` 的重連延遲（1–3 秒），但透過節流與保護機制可將影響降到最低。
