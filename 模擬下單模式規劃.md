# 模擬下單模式（Dry-Run）規劃

> 目標：開啟此模式後，**所有下單動作不送出至富邦 SDK**，但完整保留：登入、行情訂閱、策略決策、委託單流向、成交模擬、損益計算、Audit Log。
> 用於：上線前壓力測試、策略驗證、教學展示、回放比對。

---

## 一、現況評估

### 1-1 已存在的相關元件

| 元件 | 位置 | 現況 |
|---|---|---|
| `MockAdapter` | [build_win/src/broker/adapter.py](build_win/src/broker/adapter.py) | 完整模擬 broker，**完全不連 Fubon** |
| `MockOrderManager` | [build_win/src/broker/orders.py](build_win/src/broker/orders.py) | Timer 模擬延遲成交 + dispatch fill |
| `FubonAdapter.dry_run` | [build_win/src/broker/adapter.py:222](build_win/src/broker/adapter.py) | 已有 flag，預設 True |
| `FubonOrderManager.dry_run 分支` | [build_win/src/broker/orders.py:151](build_win/src/broker/orders.py) | **不完整**：只發 PENDING，沒模擬成交、沒 audit |

### 1-2 現有 dry-run 缺口
```python
# orders.py 現況
if self.adapter.dry_run:
    order_id = f"DRY{uuid.uuid4().hex[:8].upper()}"
    self.adapter.dispatch_order(OrderEvent(... PENDING ...))
    return order_id   # ← 結束。沒有 fill，沒有 log file
```

| 缺口 | 影響 |
|---|---|
| 沒有 `dispatch_fill` | 引擎以為永遠 pending，F5/F4 不會觸發、損益永遠 0 |
| 沒寫到專屬 log 檔 | 無法事後比對「如果是真單會發生什麼」 |
| GUI 沒有切換開關 | 只能改 `.env`，不直觀 |
| 沒有區分「Mock 行情 + Mock 下單」與「真實行情 + Mock 下單」 | 無法做最有價值的「真實行情壓測」 |

## 二、設計目標

### 2-1 三種運作模式

| 模式 | 行情 | 下單 | 用途 |
|---|---|---|---|
| **A. 純模擬** (`MockAdapter`) | Mock | Mock | 離線開發、單元測試 |
| **B. 真實行情 + Dry-Run** (`FubonAdapter` + `dry_run=True`) | **真實** | **Mock** | **本案目標**：上線前壓測 |
| **C. 真實實單** (`FubonAdapter` + `dry_run=False`) | 真實 | 真實 | 正式運作 |

### 2-2 Dry-Run 模式必須做到

1. ✅ 連線富邦、選擇帳號正常
2. ✅ 訂閱真實 WebSocket 行情（與實單一致）
3. ✅ 引擎判斷出進場/出場時機，產生 `OrderRequest`
4. ❌ **不呼叫** `sdk.stock.place_order()`
5. ✅ 由 `DryRunOrderManager` 接管，模擬：
   - 立即送出 `PENDING` 委託回報
   - 依設定延遲（預設 0.5~1.5s）後送出 `FILLED` 委託回報 + `FillEvent` 成交回報
   - 成交價以委託價為準（亦可選用「最新成交價」更貼近真實）
6. ✅ 寫入專屬 `dry_run_audit.jsonl`（每筆委託、成交、取消都記錄）
7. ✅ GUI 標題列、狀態列顯示醒目「🟡 模擬下單模式」紅字
8. ✅ 切換時需 GUI 二次確認，避免誤觸

## 三、技術方案

### 3-1 新增類別 `DryRunOrderManager`

**位置**：`build_win/src/broker/orders.py`

```python
class DryRunOrderManager(OrderManager):
    """
    模擬下單管理器：行為等同 MockOrderManager，但會額外寫入 audit log，
    並可選擇以即時 last_price 作為成交價（更貼近真實）。
    """

    def __init__(self, adapter, fill_delay_range=(0.5, 1.5),
                 audit_path: Optional[str] = None,
                 use_market_price: bool = False) -> None:
        self.adapter = adapter
        self.fill_delay_range = fill_delay_range
        self.use_market_price = use_market_price
        self._lock = threading.Lock()
        self._orders: dict[str, OrderRequest] = {}
        self._cancelled: set[str] = set()
        self._audit_path = audit_path or self._default_audit_path()
        self._audit_lock = threading.Lock()

    def _default_audit_path(self) -> str:
        from datetime import date
        return f"dry_run_audit_{date.today():%Y%m%d}.jsonl"

    def _write_audit(self, event_type: str, payload: dict) -> None:
        import json
        record = {
            "ts": datetime.now().isoformat(),
            "type": event_type,
            **payload,
        }
        with self._audit_lock, open(self._audit_path, "a", encoding="utf-8") as f:
            f.write(json.dumps(record, ensure_ascii=False, default=str) + "\n")

    def place_order(self, req: OrderRequest) -> str:
        order_id = f"DRY{uuid.uuid4().hex[:10].upper()}"
        with self._lock:
            self._orders[order_id] = req

        self._write_audit("PLACE", {
            "order_id": order_id, "code": req.code, "name": req.name,
            "side": req.side.value, "price": str(req.price), "qty": req.qty,
            "order_type": req.order_type, "day_trade": req.day_trade,
            "note": req.note,
        })

        self.adapter.dispatch_order(OrderEvent(
            order_id=order_id, code=req.code, side=req.side,
            price=req.price, qty=req.qty, filled_qty=0,
            status=OrderStatus.PENDING, time=datetime.now(),
        ))

        import random
        delay = random.uniform(*self.fill_delay_range)
        timer = threading.Timer(delay, self._simulate_fill, args=(order_id,))
        timer.daemon = True
        timer.start()
        return order_id

    def _simulate_fill(self, order_id: str) -> None:
        with self._lock:
            req = self._orders.get(order_id)
            if req is None or order_id in self._cancelled:
                return

        # 成交價：預設使用委託價；可選擇即時最新價
        fill_price = req.price
        if self.use_market_price:
            fill_price = self._get_latest_price(req.code) or req.price

        now = datetime.now()
        self.adapter.dispatch_order(OrderEvent(
            order_id=order_id, code=req.code, side=req.side,
            price=fill_price, qty=req.qty, filled_qty=req.qty,
            status=OrderStatus.FILLED, time=now,
        ))
        self.adapter.dispatch_fill(FillEvent(
            order_id=order_id, code=req.code, name=req.name,
            side=req.side, price=fill_price, qty=req.qty, time=now,
        ))
        self._write_audit("FILL", {
            "order_id": order_id, "code": req.code,
            "side": req.side.value, "price": str(fill_price), "qty": req.qty,
        })

    def cancel_order(self, order_id: str) -> bool:
        with self._lock:
            self._cancelled.add(order_id)
            req = self._orders.get(order_id)
        self.adapter.dispatch_order(OrderEvent(
            order_id=order_id,
            code=req.code if req else "",
            side=req.side if req else OrderSide.BUY,
            price=req.price if req else Decimal("0"),
            qty=req.qty if req else 0,
            filled_qty=0,
            status=OrderStatus.CANCELLED,
            time=datetime.now(),
        ))
        self._write_audit("CANCEL", {"order_id": order_id})
        return True

    def _get_latest_price(self, code: str) -> Optional[Decimal]:
        # 從 adapter 取得 RealtimeFeed 最新成交價
        feed = getattr(self.adapter, "_realtime_feed", None)
        if feed and hasattr(feed, "last_price"):
            return feed.last_price(code)
        return None
```

def _get_order_manager(self):

## 四、安全防護

### 4-1 防止誤切到實單
1. 預設 `dry_run=True`
2. 切到實單時 GUI 強制二次確認彈窗
3. 啟動時若 `dry_run=False`，主視窗 banner 改為**紅底白字**「⚠️ 實單模式 — 真實下單即將執行」
4. `.env` 中加入註解：`FUBON_DRY_RUN=true  # 改 false 前請確認金額與帳號`

### 4-2 防止與真實帳戶資料污染
- Dry-run 模式下，`account_service.inventories()` 仍從券商抓**真實庫存**（這對策略行為一致性很重要）
- 但 dry-run 自己的「模擬部位」不可影響真實庫存顯示
- 解法：dry-run 維護獨立的 `_dry_positions` dict，GUI 持倉表加「真實庫存 / 模擬庫存」分頁

### 4-3 風險邊界
| 情境 | dry-run 行為 |
|---|---|
| 真實帳戶有庫存（昨日留倉） | 庫存表顯示真實，但 dry-run 不會發 SELL（因 engine 不接管，目前如此） |
| 真實帳戶當天有手動下單 | 不影響 dry-run 統計（dry-run 自記自算） |
| Wifi 斷線 / 行情中斷 | dry-run 同樣會卡，不會自動切回 |

## 七、後續延伸（非必要）

- **回放模式**：把歷史行情 dump 成 jsonl，餵給 `MockRealtimeFeed`，跑完整 dry-run 流程，得到「假設策略當日跑會怎樣」的結果
- **多場景對比**：同一份行情，不同策略參數，並行跑多個 engine 實例，產出比較報告
- **Slippage 模擬**：成交價隨機 ±N tick，更貼近真實成交
- **拒單模擬**：以機率 p 模擬拒單（如餘額不足、漲停買不到），驗證引擎錯誤處理路徑

## 八、關鍵決策點（已決策）

1. **GUI 開關位置**：採用「設定」分頁，與策略參數放在同一側邊欄。
2. **成交價策略**：依建議，預設使用委託價成交；若日後要更貼近真實，可用 `FUBON_DRY_RUN_USE_MARKET_PRICE=true` 改為最新成交價。
3. **是否模擬拒單**：依建議，預設 100% 成交，不加入拒單隨機性，避免測試結果難以重現。
4. **是否需要「半模擬」**：依建議，不做部分股票實單 / 部分股票模擬，避免誤操作。

以上決策已進入實作。
